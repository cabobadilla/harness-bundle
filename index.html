01 · primer

What is harness engineering?

The architecture you place around a model — prompts, tools, permissions, hooks, memory, agents — so it can do meaningful, long-running, autonomous work.

A prompt asks the model a question. A harness turns the model into a system: one that can read your repo, run your tests, follow your conventions, hand off to specialized agents, and ship work over hours — not a single turn.

Without a harness: a chatbot. With a harness: an autonomous coding agent.

02 · the problem

50+ skills, no harness, no strategy

Most setups accumulate community plugins and skills until everything competes for the model's attention.

  • ~50 auto-triggering skills — fire whether you want them or not
  • Plugin overlap — multiple tools claim the same job
  • No CLAUDE.md contract — model reads ambient policy instead of project policy
  • Same setup for every model — scaffolding that helped Sonnet 4 now hurts Opus 4.5+

The hidden cost

Each component of a harness encodes an assumption about what the model can't do alone.

Those assumptions expire as models improve. A harness that doesn't evolve becomes a harness that fights the model.

More scaffolding ≠ more capability.
03 · the insight

Re-examine the harness with every new model

"Each harness component codifies an assumption about what the model can't do solo. Those assumptions expire as models improve."
— Anthropic Labs, Harness design for long-running application development

Sonnet 4.5

Needed context resets + sprint decomposition. Model suffered "context anxiety".

Opus 4.5

Context resets removed. Model handles long tasks natively.

Opus 4.6+

Sprints removed. Evaluator becomes conditional, not default.

"Claude Opus is our most capable model — built for complex coding, agents, and tasks that require deep, multi-step reasoning."
— Anthropic, on Claude Opus for Claude Code  ·  recommended target: Opus 4.5+
04 · the solution

Planner · Generator · Evaluator — minimal, graduated

Planner

Goal: expand a 1-4 sentence prompt into a rich, ambitious spec.

Limits: never writes source code; high-level only — over-specifying tech details propagates errors downstream.

Output: memory/specs/<slug>.md + updates to CLAUDE.md & docs/architecture.md.

Generator

Goal: implement the spec feature by feature; resolve open items in memory/backlog.md.

Limits: one feature at a time; self-evaluates before declaring done (tests + verification skill); no new deps without justification.

Output: code + conventional commits + honest updates to architecture.md when structure diverges.

Evaluator (conditional — Level B / C)

Goal: skeptical QA — find what's broken, missing, or "looks fine but isn't".

Limits: never rubber-stamps; hard thresholds per criterion. B = code/tests review (no browser). C = Playwright, browser-driven.

Output: P0/P1/P2 findings appended to memory/backlog.md + report in memory/evaluations/.

LevelAgentsWhen
APlanner + GeneratorDaily work
B+ Light evaluator (no browser)Logic, edge cases
C+ Playwright evaluatorUI deliverables

Rule of thumb: stay at A as long as you can. Move up only when a real task proves the current level isn't enough.

05 · the implementation

What harness-bundle ships

  • init-harness.sh — interactive bootstrap, idempotent, no external runtime
  • check-skills.sh — audits ~/.claude/ before & after
  • 3 agents: planner, generator, evaluator-light
  • 5 slash commands: /plan /config-stack /build /evaluate /ship
  • 3 universal skills: TDD, systematic-debugging, verification
  • 6 hooks: secret filter, bash guards, session summary, ...
  • Decoupled: shell assembles, assets/ holds content

End-to-end flow

/plan → /config-stack → /build
                       ↓
                  /evaluate  (optional)
                       ↓
                    /ship

Stack constraints

bash 3.2 POSIX no jq

Runs on macOS default shell. No Node, no Python, no external deps.
